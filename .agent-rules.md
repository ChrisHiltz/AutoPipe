### CRITICAL SYSTEM RULES FOR ALL AGENTS

You are an autonomous agent operating within the Build-Pipe document pipeline.
You must strictly obey these rules. Violations will be caught by CI and the PR will be rejected.

---

**RULE 1 — NO ORPHANED CODE:**
You may NEVER write source files in `src/` unless there is a matching PRD in `docs/04-specs/`.
If you are assigned an issue but no PRD exists, you must FIRST write the PRD using `.templates/prd_template.md`, then STOP and wait for approval.

**RULE 2 — CROSS-LINKING:**
Every document must link back to its predecessor in the pipeline chain:
- Discovery (PRB) documents must link to their raw input source (issue, signal doc, or feedback)
- Architecture (ADR) documents must contain a link to the Discovery document they solve
- Specification (PRD) documents must link to the Architecture document they implement
CI validates these links. Missing links = failed build.

**RULE 3 — STATE MACHINE:**
Do not attempt to discover, architect, AND code in the same step.
- **Discovery phase:** Read raw inputs → write `docs/02-discovery/PRB-{id}.md` → STOP
- **Architecture phase:** Read PRB → write `docs/03-architecture/ADR-{id}.md` → STOP
- **Specification phase:** Read ADR → write `docs/04-specs/PRD-{id}.md` → STOP
- **Code phase:** Read PRD → write `src/` + `tests/` → STOP
Each phase produces one type of artifact. Never skip a phase.

**RULE 4 — TECH STACK COMPLIANCE:**
Read `stack.yaml` in the project root for the mandated technology choices.
You MUST use the specified frameworks, libraries, and tools. Do not substitute, hallucinate, or invent alternatives. If `stack.yaml` does not exist or does not specify a choice for your domain, ask for clarification before proceeding.

**RULE 5 — TEMPLATE COMPLIANCE:**
When creating pipeline documents (PRB, ADR, PRD), you MUST use the templates in `.templates/`.
Fill in every section. Do not leave placeholder text like `[Short Title]` or `[YYYY-MM-DD]`.

**RULE 6 — SINGLE STAGE PER PR:**
Each Pull Request must contain artifacts from exactly ONE pipeline stage.
- A discovery PR touches only `docs/02-discovery/`
- An architecture PR touches only `docs/03-architecture/`
- A specification PR touches only `docs/04-specs/`
- A code PR touches `src/` and `tests/` (and may update the PRD checklist in `docs/04-specs/`)
Never combine discovery + architecture, or architecture + specification, in the same PR.
CI validates this rule and will reject multi-stage PRs.

**RULE 7 — BUG REPORTING:**
When you encounter a bug or issue in pipeline infrastructure — scripts, workflows, setup skills, templates, or agent instructions — you MUST report it as a GitHub Issue on the upstream template repo. Read `template.upstream_repo` from `pipeline.yaml` and run:
```
gh issue create --repo "{upstream_repo}" --title "[improvement] {component}: {short description}" --label "improvement"
```
Include the severity, affected component, what's broken, and a suggested fix if known. If `template.upstream_repo` is not set or `gh` is not authenticated, note the issue in your PR description instead. Do not silently work around bugs. Every issue discovered is an improvement opportunity.
